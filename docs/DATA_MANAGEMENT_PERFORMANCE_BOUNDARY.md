# 資料管理頁效能邊界

本文定義資料管理頁在大資料量下的主交易路徑與分析路徑邊界，避免把所有問題都丟給同一套查詢。

## 1. 目標

資料管理頁的核心責任是：

- 快速進頁
- 穩定列表瀏覽
- 即時搜尋與基礎篩選
- 添加備註與常用工具操作

這些都屬於主交易路徑，應優先由 PostgreSQL 固定欄位、索引、keyset 分頁與精簡 API 支撐。

## 2. 不該由主列表承擔的工作

下列工作在 2000 萬筆到億級時，不應直接壓在資料管理頁主列表：

- 大規模資料品質掃描
- 清理無效的全表預覽
- promotion 前後的欄位差異驗證
- 複雜統計報表
- 長時間聚合匯出前分析

這些工作應優先轉移到背景任務、快取結果或分析側車。

## 3. DuckDB 的適合位置

DuckDB 適合擔任 analytical sidecar，而不是 CRM 主列表的即時資料來源。

較適合的場景：

- 清理無效資料的批次掃描、預覽與報表
- `field-promotion` 回填後的差異比對與 validation report
- 統計分析頁、資料驗證頁
- 大型匯出前的離線資料整形

不建議直接用在：

- 資料管理頁即時列表
- 單筆編輯 / 添加備註
- 欄位管理 CRUD
- 即時匯入後立刻查詢

## 4. attrs 與固定欄位的邊界

資料管理頁若要在大資料量下維持順暢，常用欄位必須 promotion 成固定欄位。

建議原則：

- 常用搜尋欄位：固定欄位
- 常用篩選欄位：固定欄位
- 常用排序欄位：固定欄位
- 偶發、治理中的欄位：可短期留在 `attrs`

也就是說：

- `attrs` 適合承接新欄位與過渡治理
- `attrs` 不適合長期承接高頻列表操作

## 5. 主列表優先順序

若資料管理頁已出現 10 秒以上進頁延遲，優先順序應是：

1. 精簡首屏 API 與阻塞關係
2. 精簡列表回傳欄位與 `attrs` 載荷
3. 改善搜尋與篩選查詢形狀
4. 補固定欄位與索引
5. 再評估分析側車

## 6. 一句話結論

資料管理頁要順，先修 PostgreSQL 主交易路徑與固定欄位 promotion；DuckDB 應放在分析、驗證、批次掃描，不應先拿來承接主列表即時互動。

## 7. 目前導入優先順序

若之後正式導入 DuckDB，第一波優先順序如下：

1. 清理無效預覽
2. promotion validation

理由：

- 都屬於讀多寫少、規則型或驗證型工作
- 容易與主列表切開
- 成果明顯
- 與目前 roadmap 直接相關
