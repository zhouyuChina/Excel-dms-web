# 欄位管理 — 重構與優化（需求探討）

> **目的**：對齊 **資料管理** 的使用情境與體驗層次，釐清**業務必要性**，分階段落地，避免為重構而重構。  
> **關聯**：[`CRM_MASTER_BRIEF.md`](./CRM_MASTER_BRIEF.md)（專案總覽）、[`FIELD_PROMOTION_HANDOFF.md`](./FIELD_PROMOTION_HANDOFF.md)（欄位升級最新交接）。
> **狀態註記（2026-04-16）**：本文件主要描述欄位管理的 vNext 重構方向；目前右側已先落地 **待升級欄位 / `field-promotion` 側欄**，不是本文件 §8 所述的 AI 側欄。

**最後更新**：2026-04-16（移除欄位模板；**分組＝群組** 用語見 §6.0；新增 §8 AI 側欄規劃）

---

## 1. 為什麼要對齊「資料管理」？

| 資料管理（使用者） | 欄位管理（管理員／導入負責人） |
|--------------------|--------------------------------|
| 依**欄位定義**顯示表格、篩選、匯出 | 定義**名稱、型別、可見、可匯出、排序、分組** |
| 關鍵字搜尋、多條件篩選、排序 | 影響主檔 UI 與匯出欄位集合 |
| 大量資料、keyset 分頁 | 欄位數量通常 **十～數十～上百**（需快速找到欄位） |

**業務實質**：兩者是**同一資料模型的一體兩面**——沒有清楚的欄位管理，資料管理會「看得到欄位卻不知道從哪來」；沒有資料管理反饋，欄位管理容易變成純後台表單與現場脫節。

## 2. 現況差距（簡述）

| 面向 | 資料管理 | 欄位管理（重構前） |
|------|----------|---------------------|
| 脈絡說明 | 搜尋／篩選與列表一體 | 較少提示與主檔的關聯 |
| 導航 | 工具選單與多模組切換 | 與「回到資料主檔」路徑不顯眼 |
| 大量項目 | 分頁／搜尋 | 欄位多時不易定位 |
| 錯誤回饋 | 已逐步統一 `formatApiThrownError` | 多為泛用「儲存失敗」 |

## 3. 需求分層（業務是否買單）

### 3.1 P0 — 幾乎一定有業務價值

- **說清楚影響範圍**：變更欄位會影響「資料管理表格、匯出欄位、篩選下拉」。
- **一鍵回到資料管理**：減少側欄點兩次的斷裂感。
- **可讀的 API 錯誤**：與資料管理路徑一致，利於排查權限／驗證失敗。
- **欄位列表搜尋**：欄位一多時為剛需；篩選時暫停拖曳排序（避免與全域 `sortOrder` 衝突）。

### 3.2 P1 — 視組織規模與流程

- **變更稽核摘要**：誰在何時改了欄位可見性／匯出（若已有 `audit_logs` 可只讀聚合）。
- **分組**（`field-groups`，含 **顏色**）與 **資料管理 filterTags** 的命名與順序一致化（需產品用語統一）。

### 3.3 P2 — 需專案時程／產品拍板

- **欄位級權限**（誰能改系統欄、誰能新增自訂欄）。
- **與匯入對應**：別名（aliases）與表頭對應的引導精靈（見 `DEFAULT_FIELD_PREFILL_LOGIC.md`）。
- **版本化欄位定義**（回溯某一時間點欄位快照）：成本高，億級場景再談。

| 階段 | 內容 | 驗收 |
|------|------|------|

## 5. 開放問題與已定案

### 5.1 已定案（分組一鍵可見）

- **語意**：一鍵可見 **等同** 將 **預設可見性** 設為「**僅目前分組內欄位可見，其餘欄位不可見**」— 在體感上接近 **套用一條「只看這一包」的條件**，只是寫入的是欄位定義上的 **`defaultVisible`**，不是資料管理裡對**列**的篩選。  
- **可匯出**：**不納入**一鍵操作；**`isExportable` 維持不變**，匯出仍依既有欄位定義與匯出流程（若要匯出某批欄位，使用者到資料管理再開匯出即可）。  
- **貼心提醒（非阻斷）**：套用成功後，可顯示輕量提示，例如：「**接下來要匯出嗎？**」並提供 **前往資料管理**（或若產品允許，深鏈至匯出／匯出精靈入口），**不強制**、不打斷流程。

### 5.2 尚待釐清（請產品／業務填）

1. **誰有權**進入欄位管理？與 `editor`／`admin` 是否一致？  
2. **自訂欄**上限與命名規則（避免與未來系統欄衝突）？  
3. **治理指標**（非空比例、最後有值時間）可接受的計算方式：即時查詢、每日批次、或取樣近似？  
4. **合併欄位**是否一律從治理列表進入，或列內即可發起？

## 6. vNext 整體定位與資訊架構（合併稿）

本節整合討論定案方向：**動態欄位持續增長**時，欄位管理同時承担 **CRUD**、**依情境批次調整**、**治理（去重／幽靈欄）**；並以 **與資料管理同一掃視節奏** 為 UI 目標。

### 6.0 用語：分組、群組、顏色

- **後端／API** 使用 **`field-groups`**（既有不變）。  
- **產品介面**以 **「分組」** 為主（與現有「分組管理」一致）。  
- 文件中的 **「群組」與「分組」同義**，指同一實體。  
- **設計原意**：分組帶 **顏色**，讓使用者在 **資料管理欄位標籤／篩選** 時能 **靠色塊直覺點選**；vNext 的 **分組 chip、依分組一鍵預設可見** 與此一致，並強化「情境包」操作，**不再另做「欄位模板」功能**（已自前端與 API 客戶端移除；見 §6.5）。

### 6.1 心智模型

| 層級 | 角色 |
|------|------|
| **資料管理** | 在**已定義**的欄位上工作（列、儲存格、篩選、匯出）。 |
| **欄位管理** | 管**欄位定義**本身：新增／編輯／刪除、可見與匯出規則、排序與分組；並在動態場景下提供 **資料管理做不到** 的 **治理與情境批次** 能力。 |

**獨有價值（與資料管理互補）**

- **治理**：疑似**重複**（合併候選）、**整欄無資料或極低使用率**（動態掃入的幽靈欄）— 需列表級 **訊號＋篩選**，必要時接既有 **合併／刪除** 流程。  
- **情境**：**分組**可對應「匯出包／協作包」；在已篩出該分組欄位的前提下，提供 **一鍵調整預設可見性**（**不**改可匯出；見 §5.1、§6.4）。

### 6.2 單一主戰場：欄位列表 + 工具列（含 CRUD）

**預設 landing**：**欄位**列表為全頁核心；**新增／編輯／刪除** 不另開平行資訊架構，而集中在同一視線層級。

| 能力 | UI 建議 | 說明 |
|------|---------|------|
| **新增** | 頂部主要按鈕 **「新增欄位」**（與搜尋、chip 同區） | 開啟 **表單**（modal 或右側抽屜）；送出後回到列表並捲動／聚焦新列可選。 |
| **編輯** | 列上 **「編輯」** 或列點擊 → **抽屜為主** | **高頻欄位**（名稱、型別、可見、可匯出、分組、別名）在抽屜內一次完成；與資料管理密度一致時，列表可保留 **輕量 inline**（例如可見／匯出切換）視實作成本取捨。 |
| **刪除** | 列內次要動作 **「刪除」** | **確認對話框**；系統欄、有依賴時禁用或說明原因；與 **合併** 分開，避免誤触。 |
| **合併** | 治理 **badge** 或列動作 **「合併…」** | 與既有 **MergeFieldsModal／高風險** 流程一致；可從「疑似重複」篩選結果批次進入（Phase C）。 |

**拖曳排序**：僅在 **未套用會與全域 `sortOrder` 衝突的篩選** 時啟用（既有行為）；工具列可一句話標示「目前排序依：手動」。

### 6.3 分組：chip 篩選 + 需要時才「管理分組」

- **日常**：頂部 **分組 chip**（`全部`、`未分組`、各分組名，**顏色**與資料管理標籤呼應）— **僅篩選列表**，不離開欄位頁。  
- **維護分組 CRUD**：chip 列末端 **「管理分組…」**（或維持現有 **分組管理** 分頁）→ 內做 **新增／改名／顏色／排序／刪除**。  
- **指派欄位所屬分組**：在 **編輯抽屜** 內下拉為主；必要時在列表加 **分組** 欄只讀顯示以利掃視。

### 6.4 情境：依分組一鍵調整預設可見性

在已選定某 **分組 chip**（列表已篩成該情境包）時，工具列提供 **「套用此分組可見性」** 等按鈕（文案可依產品再調）。

- **行為（已定案）**：將 **屬於該分組** 的欄位設為 **預設可見**，**其餘欄位** 設為 **預設不可見** — 語意上等同 **套用一條「只亮這一包」的條件** 到 **`defaultVisible`**；**不**修改 **`isExportable`**。  
- **用途**：月底對帳、活動檔期、跨部門協作等 **短期對齊主檔預設顯示**；若接著要匯出，仍到 **資料管理** 依既有匯出流程操作。  
- **貼心提醒**：套用成功後可顯示 **非阻斷** 提示：「**接下來要匯出嗎？**」並附 **前往資料管理**（或產品允許時之匯出入口）；見 §5.1。

### 6.5 已移除：欄位模板

- **決策**：與 **分組一鍵預設可見** 價值重疊，且產品希望以 **分組（顏色／情境）** 為單一治理語彙，故 **不再提供「欄位模板」** 獨立功能。  
- **程式**：已移除 `FieldTemplateTab`、`/api/field-templates` 註冊與 `dmsApi` 對應方法；若需 **跨分組** 的具名預設，留待未來另議（例如「欄位配置集」），不復用舊模板 UI。

### 6.6 治理：列表訊號與篩選（Phase C 為主）

- **列表加欄（或 badge）**：**非空比例**／**有值筆數**、**疑似與他欄重複**（啟發式：名稱、alias、型別、群組）。  
- **篩選**：僅 **低使用率**、僅 **疑似重複**、僅 **自訂欄** 等，讓動態新增下 **問題欄位先浮上來**。  
- **前提**：全表統計需 **後端聚合或批次**（見第 5 節 §5.2 問題 3）；第一階段可先做 **純前端／啟發式重複**。

### 6.7 UI/UX 最佳化方案（原則）

**與資料管理對齊**

- **橫向工具列**：脈絡一句話 + **前往資料管理** + **搜尋** + **分組 chip**（可橫向捲動）+ **新增** + **管理分組**；必要時第二列放 **治理／可見／可匯出／系統欄** 篩選 chip。  
- **密度**：主區以 **緊湊表格／虛擬列表** 為預設；**大卡片** 不作預設列視圖，避免與「上百欄位」掃視目標衝突。  
- **編輯**：**右側抽屜** 承載完整表單；列表只保留 **最少欄位 + 狀態 + 操作**，降低視覺重量。

**狀態與信任**

- **載入**：列表骨架；**空狀態**（無欄位／篩選無結果）分開文案。  
- **高風險**：刪除、合併、一鍵大範圍改 **預設可見** — **確認前摘要影響**（主檔預設顯示欄位；一鍵**不**改可匯出設定）。  
- **錯誤**：延續 **Phase A** 的 `formatApiThrownError` 一致體驗。

**頂層分頁**

- 保留 **欄位管理 | 分組管理**（分組 chip 進階可收斂為「管理分組」第二層，見 Phase B）；**不再**提供「欄位模板」分頁。

**驗收一句話（vNext）**

- 使用者可在 **同一欄位頁** 完成 **搜尋、依分組篩選、新增／編輯／刪除、**（Phase C）**治理篩選與合併入口、** **依分組一鍵調整預設可見性**（其餘欄位不可見；見 §5.1）**，且視覺節奏與 **資料管理** 同一家族。

## 8. 未來實作區：右側 AI 側欄（規劃）

本節先定義 **欄位管理頁** 的右側 AI 側欄方向；現階段僅做規劃，不改既有主流程。

### 8.1 目標與角色

- **定位**：欄位管理的「治理助理」，先做 **洞察與建議**，再做 **人工確認後套用**。  
- **範圍**：服務欄位定義層（重複、命名、分組、可見性建議）；不直接改動資料列內容。  
- **原則**：預設 **只讀**；所有寫入都需使用者確認。

### 8.2 側欄資訊架構（建議）

| 區塊 | 內容 | 初期模式 |
|------|------|----------|
| **洞察（Insights）** | 疑似重複欄位、未分組欄位、命名異常、低使用率（若有統計） | 規則式（前端） |
| **建議（Suggestions）** | 建議合併、建議分組、建議可見性調整 | 規則式 + 可人工勾選 |
| **問答（Ask AI）** | 自然語言詢問欄位治理問題，附理由與依據 | 後續接 AI |
| **待套用清單（Review Queue）** | 多條建議預覽、批次套用、套用記錄 | 人工確認後寫入 |

### 8.3 互動規則（安全邊界）

- **不可靜默寫入**：AI 產生的操作需先進入待套用清單。  
- **顯示理由**：每條建議附判斷依據（名稱相似度、別名重疊、分組一致性等）。  
- **顯示影響**：套用前顯示將變更的欄位與欄位屬性（diff）。  
- **可撤回**：至少支援同 session 撤回；中期接 `audit_logs` 做可追溯。  
- **權限一致**：沿用欄位管理頁權限策略（見 §5.2）。

### 8.4 分階段落地（建議）

| 階段 | 內容 | 依賴 |
|------|------|------|
| **AI-Phase 0（UI 骨架）** | 右側可收合側欄、四區塊占位、與主列表同步選取上下文 | 前端 |
| **AI-Phase 1（規則建議）** | 規則式洞察與建議（無模型）、待套用清單、人工確認套用 | 前端 + 既有 patch API |
| **AI-Phase 2（語意建議）** | 接 AI 問答與語意重複判斷、顯示置信度與理由 | 後端 AI gateway |
| **AI-Phase 3（治理閉環）** | 套用稽核、撤回、批次任務化與狀態追蹤 | `audit_logs` / `jobs` |

### 8.5 與現有 UI 的銜接點

- 主畫面維持 **單一主面板**（新增可摺疊 + 工具列 + 列式編輯）。  
- AI 側欄不取代分組 chip/欄位標籤；僅補上「建議與治理」能力。  
- 欄位標籤與分組狀態可作為 AI 建議上下文（例如「目前只看某分組」）。

### 8.6 驗收（規劃版）

- 使用者可在不離開欄位頁的情況下查看 AI/規則建議。  
- 每條建議皆可檢視理由、選擇是否納入待套用清單。  
- 套用前有變更預覽，套用後有回饋與可追溯紀錄。  
- 若 AI 模組不可用，欄位管理主流程不受阻（可降級為規則模式）。
